# 双创作界面统一接入执行清单

## 1. 目标

把当前项目整合成下面的最终形态：

- 一个后端
- 一个用户系统
- 一个管理后台
- 两个创作界面
- 同域名不同路径切换

并明确本次只做下面这些共享：

- 共享登录
- 共享点数
- 共享模型
- 共享线路
- 共享订单

本次明确不做强共享：

- 不共享画布草稿
- 不共享画布节点状态
- 不共享经典版本地临时历史
- 不共享经典版本地参考图缓存

## 2. 已确认的现状

### 2.1 当前主项目

- 主项目是 `Vite + React + Express`。
- 后端统一托管 `dist`，未知前端路径最终都会回到同一个 `dist/index.html`。
- 当前已经有统一的：
  - 登录系统
  - 点数系统
  - 管理后台
  - 模型/线路配置中心
  - 图片/视频生成网关

### 2.2 第二个经典界面项目

来源路径：

- `D:\网站开发\qq.com3-20上线\project`

现状特征：

- 是一个轻量 `Express + 静态 public 页面` 项目
- 前端是 `index.html + style.css + script.js`
- 自带旧代理后端 `server.js`
- 目前依赖用户自己输入 API Key
- 模型/线路很多逻辑写死在前端脚本里
- 历史记录、待处理任务、API Key 都主要保存在浏览器本地

## 3. 推荐的最终架构

### 3.1 路径设计

建议统一成：

- `/create/canvas`：画布版
- `/create/classic`：经典版
- `/admin`：后台
- `/`：根据用户偏好跳转，或默认进 `/create/canvas`

### 3.2 后端真相源

正式上线后，只保留当前主项目作为唯一后端。

第二个项目的 `server.js` 不再作为生产后端使用，只保留它的前端页面、样式和部分交互逻辑作为迁移素材。

### 3.3 共享与隔离原则

共享：

- 用户登录态
- 用户角色
- 点数余额
- 点数账本
- 模型目录
- 线路目录
- 生成任务提交
- 订单/任务记录

隔离：

- 画布版 `canvasStore`
- 画布版 `selectionStore`
- 经典版 `nb_history`
- 经典版 `nb_pending_tasks`
- 经典版本地参考图缓存

也就是说：

- 两个界面共用账号和业务
- 两个界面各自保留自己的本地工作态

## 4. 本次最适合的接入方式

## 4.1 选择：经典版作为“同仓库 legacy 子应用”

不要第一步就把经典版重写成 React。

最快落地方式是：

- 把经典版静态资源并入当前仓库
- 挂到当前站点同域名下
- 让它改用当前主项目后端

建议目录形态：

```text
main.aittco.com/
  public/
    classic-app/
      index.html
      style.css
      script.js
      promptTags.js
      wechat.png
```

这样做的好处：

- 视觉和交互可以最大程度保留
- 不需要维护第二套生产后端
- 登录可直接复用当前同站点 session
- 上线风险低

## 4.2 不建议的做法

暂时不建议：

- 把经典版单独放在另一个域名
- 另起一套登录体系
- 经典版继续使用它自己的 `server.js`
- 一开始就把经典版全量重写成 React

## 5. 需要改造的核心点

## 5.1 路由与托管层

### 目标

让主站支持两个创作入口，同时仍由一个后端托管。

### 需要做的事

1. 当前主项目新增前端入口规划：
   - `/create/canvas`
   - `/create/classic`
2. 后端在 React catch-all 之前，单独处理经典版入口：
   - `/create/classic`
3. 经典版静态资源使用固定前缀：
   - `/classic-app/...`
4. React 主入口保留：
   - `/create/canvas`
   - `/admin`
   - `/`

### 原因

当前后端最后有统一 catch-all，所以经典版如果直接走 `/create/classic` 而没有专门分流，会被错误地回退到 React 的 `index.html`。

## 5.2 认证层

### 目标

两个界面使用同一个登录系统。

### 需要做的事

1. 经典版移除“自己作为主身份来源的 `nb_key` 模式”
2. 经典版优先读取当前主项目 session
3. 经典版调用当前认证接口：
   - 读取当前 session
   - 获取用户信息
   - 退出登录
4. 如果你仍要兼容旧 API Key 模式：
   - 只作为补充解锁模式
   - 不再作为经典版唯一登录方式

### 最终判定逻辑

- 已登录：展示当前用户可用模型
- 未登录但输入有效旧 Key：只展示允许旧 Key 使用的模型
- 未登录且无有效 Key：不展示模型，不允许生成

## 5.3 模型与线路层

### 目标

经典版不再维护自己写死的模型和线路。

### 需要做的事

1. 删除经典版页面里写死的模型列表依赖
2. 删除经典版页面里写死的线路一/二/三依赖
3. 改为读取当前主项目公开 catalog
4. 经典版模型下拉、线路下拉，全部由后台配置驱动

### 结果

以后后台新增模型/线路：

- 画布版自动可见
- 经典版自动可见

无需在两个前端分别手工改一遍。

## 5.4 生成接口层

### 目标

经典版所有生成都走当前主项目统一网关。

### 需要做的事

1. 经典版废弃旧代理后端：
   - `/api/generate`
   - `/api/check/:id`
   - `/api/gemini-generate`
   - `/api/video/generate`
   - `/api/video/check/:id`
2. 经典版改为对接当前主项目现有生成接口
3. 经典版提交时带上：
   - `modelId`
   - `routeId`
   - `uiMode=classic`
4. 画布版提交时也补：
   - `uiMode=canvas`

### 目的

以后后台就能统计：

- 哪个界面更常用
- 哪个界面转化更高
- 哪个界面失败率更高

## 5.5 点数与订单层

### 目标

共享点数，也共享订单。

### 现实情况

当前主项目已经有统一账本和待处理任务体系，但“前台用户可直接查看的统一订单列表”还需要单独整理成正式接口。

### 本次建议补一层

新增统一“订单/任务列表”能力：

- `GET /api/account/orders`

返回内容建议包含：

- 订单/任务 ID
- 用户 ID
- `uiMode`
- 模型 ID
- 线路 ID
- 媒体类型
- 提示词摘要
- 提交时间
- 状态
- 扣点
- 输出结果 URL

### 界面侧处理

- 画布版可以新增“我的订单”
- 经典版保留自己本地“历史记录”体验
- 同时新增一个“云端订单”区域，读取统一订单接口

这样就满足：

- 共享订单
- 但不强制共享草稿

## 5.6 后台层

后台继续保留现在这一套，不新开第二后台。

后台仍然只做：

- 用户管理
- 点数管理
- 兑换码管理
- 模型管理
- 线路管理
- 运行统计

只是运行统计后面建议增加：

- `uiMode=canvas`
- `uiMode=classic`

维度。

## 7. 推荐分阶段实施

## Phase 0：冻结方案

目标：

- 确认使用“同仓库 legacy 子应用”方案
- 确认路径命名
- 确认共享范围

输出：

- 本文档确认版

## Phase 1：挂出经典版入口

目标：

- 经典版页面先能通过 `/create/classic` 打开
- 但暂时不接业务逻辑

任务：

1. 复制经典版静态资源到主项目
2. 修正资源路径为 `/classic-app/...`
3. 后端增加 `/create/classic` 入口路由
4. 首页或头像菜单增加“画布版 / 经典版”切换入口

验收：

- 访问 `/create/classic` 能看到经典版页面
- 访问 `/create/canvas` 能看到画布版页面
- `/admin` 不受影响

## Phase 2：接统一认证

目标：

- 经典版用当前主项目登录系统

任务：

1. 经典版接入当前 session 获取逻辑
2. 未登录时显示“去登录 / 输入兼容 Key”
3. 已登录后展示用户身份和点数信息
4. 移除经典版前台管理员 Key 判断

验收：

- 两个界面登录态一致
- 一个界面退出，另一个界面也视为退出

## Phase 3：接统一模型/线路目录

目标：

- 经典版模型和线路来自后台配置

任务：

1. 经典版下拉改为读取公开 catalog
2. 兼容旧 Key 的模型只在旧 Key 有效时展示
3. 需要登录的模型在未登录时不展示

验收：

- 后台新增模型/线路后，两个界面都能看到

## Phase 4：接统一生成网关

目标：

- 经典版正式接当前后端生成接口

任务：

1. 经典版废弃旧 `server.js`
2. 图片生成改接当前统一图片接口
3. 视频生成改接当前统一视频接口
4. 统一传 `modelId + routeId + uiMode`

验收：

- 经典版可以正常生图/生视频
- 点数扣减与当前主系统一致
- 后台可看到经典版请求

## Phase 5：接统一订单

目标：

- 用户在两个界面里都能看到同一套云端订单

任务：

1. 后端新增订单列表接口
2. 订单记录补充 `uiMode`
3. 经典版新增“云端订单”区
4. 画布版也可补“我的订单”

验收：

- 画布版生成的任务，经典版可看到
- 经典版生成的任务，画布版可看到
- 但各自本地草稿仍不互通

## Phase 6：体验补强

目标：

- 更像完整产品，而不是两页拼接

任务：

1. 记住用户上次使用的创作界面
2. 在账号菜单中切换创作模式
3. 在后台统计里增加 `canvas/classic` 维度
4. 做创作入口统一文案和统一头部

## 8. 最小可行版本定义

如果要尽快上线，最小可行版本应满足：

- `/create/canvas` 可用
- `/create/classic` 可用
- 共用登录
- 共用点数
- 共用模型/线路
- 共用订单
- 共用后台
- 不共享草稿

只要做到这些，就已经符合你本次的目标。

## 9. 风险与处理方式

### 风险 1：经典版继续依赖旧后端逻辑

处理：

- 生产上彻底下线第二项目 `server.js`
- 只保留页面资源

### 风险 2：经典版仍写死模型和线路

处理：

- 第一轮就改成 catalog 驱动
- 不允许保留双份模型真相源

### 风险 3：用户误以为“历史记录”已经共享

处理：

- 前台明确区分：
  - 本地历史
  - 云端订单

### 风险 4：`/create/classic` 被 React catch-all 吃掉

处理：

- 后端显式优先路由
- 放在 catch-all 之前

## 10. 我建议的实际执行顺序

按这个顺序最稳：

1. 先把经典版静态资源迁入当前仓库
2. 先挂通 `/create/classic`
3. 再接统一登录
4. 再接统一模型/线路
5. 再切换到统一生成接口
6. 最后补共享订单

不要反过来先做订单或先做重写，否则很容易范围失控。

## 11. 下一步进入开发时的工作方式

进入实际开发时，建议我按下面顺序直接执行：

1. 建立 `public/classic-app/`
2. 让 `/create/classic` 能打开页面
3. 把经典版从旧 `server.js` 中剥离出来
4. 接入统一登录状态
5. 接入统一 catalog
6. 接入统一生成接口
7. 新增统一订单接口
8. 补前台切换入口

## 12. 本次方案的最终决定

本次采用：

- 当前主项目作为唯一主仓库
- 当前主项目作为唯一后端
- 当前主项目作为唯一后台
- 第二个项目只作为经典界面静态资源来源
- 经典版优先以 legacy 子应用形式接入
- 先实现共享登录、点数、模型、线路、订单
- 不做草稿与画布状态共享

这就是当前阶段最快、风险最低、最适合上线的方案。
